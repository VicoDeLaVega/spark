6.0 Vue d'ensemble — Pourquoi la performance est-elle si délicate ?

Un kernel CUDA peut être correctement écrit et pourtant atteindre moins de 10% du débit théorique d'un GPU. Les goulots d'étranglement viennent de trois sources principales : l'accès mémoire sous-optimal (coalescing, latence DRAM), la contention de banques SRAM (bank conflicts), et le manque d'occupancy ou d'ILP. Ce chapitre démonte chaque mécanisme avec des chiffres réels sur A100.

Hiérarchie mémoire GPU — rappel quantifié (A100 SXM4)

Registres : ~255 par thread, 0 latence. Shared memory : 164 KB/SM, ~5 cycles. L1/Tex : ~32 KB, ~30 cycles. L2 : 40 MB, ~200 cycles. HBM2e (global) : 80 GB, ~600-800 cycles, 2000 GB/s.

6.1 Memory Coalescing — Mécanique Complète

Le contrôleur mémoire d'un GPU moderne opère en transactions de 32 ou 128 bytes. Quand un warp émet 32 load/store de 4 bytes, le hardware regarde si les adresses appartiennent à la même ligne de cache (128 B). Si oui → 1 transaction. Si non → jusqu'à 32 transactions distinctes.

6.1.1 Coalescing en pratique : Row-Major vs Column-Major

La règle fondamentale : thread i doit accéder à l'élément i (ou i*stride avec stride=1). La confusion classique vient des matrices 2D.

// ✅ COALESCED — thread (row,col) accède à A[row][col]
// En mémoire linéaire : A[row*N + col]
// Thread blockDim.x=32, threadIdx.x varie → col consécutif
__global__ void matTranspose_good(float* A, float* B, int N) {
    int col = blockIdx.x * blockDim.x + threadIdx.x;  // varie rapidement
    int row = blockIdx.y * blockDim.y + threadIdx.y;  // varie lentement
    if (row < N && col < N)
        B[col * N + row] = A[row * N + col];  // lecture coalesced ✅
    // (écriture non-coalesced → résolu avec shared memory, voir ch.5)
}

// ❌ NON-COALESCED — stride = N entre threads consécutifs
__global__ void columnAccess_bad(float* A, int N) {
    int row = blockIdx.x * blockDim.x + threadIdx.x;
    // Thread 0 : A[0], Thread 1 : A[N], Thread 2 : A[2N] → stride N !
    float val = A[row * N + 0];  // accès colonne → 32 transactions/warp
}
⚠️ Règle : dans un kernel 2D, threadIdx.x doit indexer la dimension la plus rapide (colonne en C/CUDA row-major). Si threadIdx.x indexe les lignes, chaque warp génère N transactions au lieu de 1.

6.1.3 Coalescing avec des types larges

// float4 : charge 4 floats en 1 instruction → meilleur ILP et débit
__global__ void saxpy_float4(float* x, float* y, float a, int n) {
    int i = (blockIdx.x * blockDim.x + threadIdx.x) * 4;
    if (i + 3 < n) {
        float4 xi = *(float4*)(x + i);  // 128-bit load coalesced
        float4 yi = *(float4*)(y + i);
        yi.x = a*xi.x + yi.x;  yi.y = a*xi.y + yi.y;
        yi.z = a*xi.z + yi.z;  yi.w = a*xi.w + yi.w;
        *(float4*)(y + i) = yi;
    }
}
// Impact : réduit le nombre d'instructions d'adressage de 4x
//          la bande passante reste identique mais le pipeline est mieux utilisé

6.2 Shared Memory Bank Conflicts — Anatomie Complète

La shared memory est organisée en 32 banques (sur GPU >=Fermi), chacune de 4 bytes de large. L'adresse byte addr appartient à la banque (addr/4) % 32. Si plusieurs threads d'un warp accèdent à la même banque (mais des adresses différentes), les accès sont sérialisés.

6.2.1 Patterns de Conflits — Exemples Concrets

// Shared mem : float smem[32][32];  → chaque ligne = 32 floats = 32 banques
// Accès par threadIdx.x :

// ✅ Stride 1 — aucun conflit
// T0→smem[0][0] (bank 0), T1→smem[0][1] (bank 1), ..., T31→smem[0][31] (bank 31)
float a = smem[threadIdx.y][threadIdx.x];  // ✅ parfait

// ❌ Stride 2 — 2-way conflict
// T0→bank 0, T1→bank 2, T2→bank 4, ..., T16→bank 0 ← CONFLIT avec T0 !
float b = smem[threadIdx.x][threadIdx.x * 2 % 32];  // ❌ 2-way

// ❌ Stride 32 — 32-way conflict (LE pire cas)
// Tous les threads accèdent à la banque 0 (rows différentes mais bank identique)
float c = smem[threadIdx.x][0];  // ❌ 32-way conflict : banque 0 pour tous

// ✅ Transposition via padding — résout le conflit classique
// Ajout d'une colonne fantôme : float smem[32][33]
// smem[i][j] → bank (i*33 + j) % 32 → tous différents !
__shared__ float tile[32][33];  // +1 colonne de padding ← LA solution
tile[threadIdx.y][threadIdx.x] = d_in[...];
__syncthreads();
float val = tile[threadIdx.x][threadIdx.y];  // ✅ accès transposé sans conflit

6.2.2 Pourquoi le Padding Fonctionne

Calcul du numéro de banque

Banque(smem[i][j]) = (i * COLS + j) % 32. Avec COLS=32 : smem[0][0]=bank 0, smem[1][0]=bank 0 aussi (car 32%32=0). → Conflit sur colonne 0.

Avec COLS=33 : smem[0][0]=bank 0, smem[1][0]=bank 33%32=bank 1 → pas de conflit ! Le padding décale chaque ligne d'une banque.

6.2.3 Detect bank conflicts avec Nsight

# Profiling CLI — cherche shared_load_transactions_per_request
# Une valeur > 1 indique un bank conflict de ce factor
$ ncu --metrics l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_ld.sum       ./my_kernel
# Output exemple :
# l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_ld.sum : 1024
# → 1024 conflits sur les loads shared → optimiser avec padding

6.3 Occupancy — Maximiser l'Utilisation des SMs

L'occupancy = warps actifs / warps max théoriques par SM. Sur A100 : 64 warps max par SM. L'occupancy permet de masquer la latence mémoire en gardant d'autres warps prêts à s'exécuter pendant qu'un warp attend des données.

Occupancy = min(warps_register_limit, warps_smem_limit, warps_block_limit) / max_warps_per_SM

void myKernel(...) { ... }
// Le compilateur alloue moins de registres pour respecter la contrainte

6.3.2 Cas Pratique : GEMM 16x16 tiles

Exemple A100 — kernel GEMM avec smem

Block 16×16 = 256 threads = 8 warps. Smem par block = 2 × 16×16 × 4 bytes = 2 KB. SM peut héberger 164K/2K = 82 blocks théoriques, limité à 32 blocks max → 32×8 = 256 warps → occupancy = 256/64 = 400%? Non, 64 warps max → 64/64 = 100%.

Mais si chaque thread utilise 128 registres : 256 threads × 128 regs = 32768 regs. A100 a 65536 regs/SM → 2 blocks max → 2×8=16 warps → 25% occupancy.

6.4 Thread Coarsening — Quand et Comment

Le thread coarsening attribue plusieurs éléments de sortie à un seul thread. On augmente l'arithmetic intensity et on amortit le coût du démarrage (prologue/épilogue de warp), au prix d'une réduction du parallélisme.

// AVANT : 1 thread → 1 élément
__global__ void kernel_v1(float* A, float* B, float* C, int N) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    C[i] = A[i] + B[i];  // 1 FLOP, 8 bytes lus → AI = 0.125 FLOP/byte
}

// APRÈS : 1 thread → 4 éléments (COARSENING_FACTOR = 4)
__global__ void kernel_v2(float* A, float* B, float* C, int N) {
    int base = (blockIdx.x * blockDim.x + threadIdx.x) * 4;
    // Registres : ai[4], bi[4] — réutilisés, moins d'overhead
    float a0=A[base], a1=A[base+1], a2=A[base+2], a3=A[base+3];
    float b0=B[base], b1=B[base+1], b2=B[base+2], b3=B[base+3];
    C[base]=a0+b0; C[base+1]=a1+b1; C[base+2]=a2+b2; C[base+3]=a3+b3;
    // 4 FLOP, 32 bytes lus → AI = 0.125 FLOP/byte (identique ici !)
    // Bénéfice : moins de blocs lancés → moins d'overhead scheduler
}
⚠️ Le coarsening aide principalement quand : (1) le kernel est limité par l'overhead de lancement, (2) l'AI augmente réellement (ex: convolution avec poids partagés), (3) le GPU est déjà quasi-saturé en warps (occupancy > 50%). Il peut réduire les performances si le GPU manque de travail (trop peu de blocs actifs).

6.5 Le Roofline Model — Guide Complet

Le Roofline model prédit la performance maximale atteignable d'un kernel en fonction de son arithmetic intensity (AI). Il prend en compte deux plafonds : la bande passante mémoire et le débit de calcul.

Performance [GFLOP/s] = min(Peak_FLOPS, Memory_BW × AI)

6.5.2 Calculer l'AI d'un Kernel

// Exemple : GEMM M×N×K naïf (sans cache)
// FLOPS = 2 × M × N × K  (multiply-add pour chaque élément de C)
// Bytes = (M×K + K×N + M×N) × sizeof(float)
// Pour M=N=K=1024 :
//   FLOPS = 2 × 1024³ = 2.1 × 10⁹
//   Bytes = 3 × 1024² × 4 = 12.6 × 10⁶ bytes
//   AI_naif = 2.1G / 12.6M ≈ 166 FLOP/byte  ← compute-bound?
//
// MAIS le naïf relit A et B K fois depuis DRAM (pas de cache L2 parfait) :
//   Bytes_réels ≈ K × (M × 4 + N × 4) = 1024 × 8K bytes = 8 GB
//   AI_réel ≈ 2.1G / 8G ≈ 0.26 FLOP/byte ← memory-bound !
//
// Avec tiling (tile=32) : chaque tile lue 1 fois en smem, K/32 times :
//   Bytes_réels ≈ (M×K/32 + K×N/32) × 4 = M×N × 4/8 ≈ 0.5 GB (simplifié)
//   AI_tiled ≈ 2.1G / 0.5G ≈ 4× → compute-bound sur A100 (ridge=9.75)

6.6 Prefetching et Pipelining Mémoire

Les GPUs utilisent le multithreading massif pour cacher la latence mémoire : pendant qu'un warp attend des données (300-700 cycles), le scheduler exécute d'autres warps. Mais quand l'occupancy est faible, cette stratégie échoue. Le prefetching logiciel peut aider.

// Double-buffering : pipeline calcul et chargement avec 2 tiles
// Pendant qu'on compute sur tile A, on charge tile B dans le background
__global__ void gemm_pipelined(...) {
    __shared__ float tileA[2][TILE][TILE];  // double buffer
    __shared__ float tileB[2][TILE][TILE];
    int cur = 0, nxt = 1;

    // Préchargement du premier tile
    loadTile(tileA[cur], A, 0);
    loadTile(tileB[cur], B, 0);
    __syncthreads();

    for (int k = 0; k < K/TILE; k++) {
        if (k + 1 < K/TILE) {
            loadTile(tileA[nxt], A, k+1);  // load prochaine tile
            loadTile(tileB[nxt], B, k+1);  // pendant le calcul
        }
        computeTile(tileA[cur], tileB[cur], C);  // compute tile courante
        __syncthreads();
        cur ^= 1; nxt ^= 1;  // swap buffers
    }
}
🎯 Questions d'Entretien NVIDIA/AMD/Intel — Ch.6
  • « Expliquez précisément pourquoi un accès stride-2 génère 2 transactions mémoire par warp. » → 32 threads × stride 2 × 4B = 256B couvrant 2 lignes de 128B → 2 transactions.
  • « Comment résoudre un bank conflict dans une transposition de matrice ? » → Padding +1 colonne : float smem[32][33] décale chaque ligne d'une banque.
  • « Qu'est-ce que le ridge point du Roofline model ? » → AI à laquelle BW×AI = Peak FLOPS. Kernels en dessous : memory-bound. Au-dessus : compute-bound.
  • « Donnez l'AI d'un GEMM naïf vs tuilé. » → Naïf ~0.25 FLOP/byte (relecture depuis DRAM), tuilé ~AI=tile_size/2 FLOP/byte. Tile=32 → AI≈16, compute-bound sur A100.
  • « Quand le thread coarsening nuit-il ? » → Quand le nombre de blocs/SM devient inférieur à 2-4 (latency hiding insuffisant), l'occupancy chute et la latence DRAM n'est plus cachée.
  • « Comment mesurez-vous le débit mémoire effectif d'un kernel ? » → ncu --metrics l1tex__t_bytes_pipe_lsu_mem_global_op_ld.sum donne les bytes transférés. Débit = bytes / temps kernel.
