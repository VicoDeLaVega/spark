Spark — Data & GPU Pipeline

Comment les splats passent du disque aux shaders WebGL

Pipeline principal
1

URL / File bytes

Point d'entrée. Les données arrivent soit via une URL HTTP, soit comme bytes bruts passés directement.

Pour les gros fichiers, des Range requests HTTP sont utilisées pour charger chunk par chunk (SplatPager.ts:186).

2

SplatLoader — entry point

loadInternal() reçoit l'input (URL, bytes, ou stream) et délègue immédiatement à un pool de workers via RPC pour ne pas bloquer le thread principal.

SplatLoader.ts:198 — worker.call()
3

Worker Thread — WASM Decoder

Le worker streame le fichier par chunks de 1 MB, les pousse au décodeur WASM, et récupère le résultat final.

Deux formats de sortie possibles : loadPackedSplats (Uint32, 16 bytes/splat) ou loadExtSplats (Float32, pour les coordonnées étendues).

worker.ts:242 — loadPackedSplats()

4

PackedSplats — stockage CPU

Le résultat décodé est stocké dans this.packedArray: Uint32Array. C'est le buffer principal en RAM — un tableau plat de 16 bytes par splat.

Les coefficients de Spherical Harmonics (pour l'éclairage directionnel) sont dans des tableaux Float32 séparés : sh1, sh2, sh3.

5

⚡ Upload GPU — le moment clé

maybeUpdateSource() crée une THREE.DataArrayTexture avec les bytes du packedArray, puis pose le flag magique :

texture.needsUpdate = true

C'est THREE.js qui détecte ce flag au prochain frame et appelle l'API WebGL sous-jacente (texImage3D) pour uploader les données en VRAM. Pas de .to(device) comme en PyTorch — tout est implicite via ce flag.

6

GPU — DataArrayTexture RGBA32UI

Les splats sont maintenant en VRAM sous forme d'une texture 3D (Array Texture WebGL 2). Format RGBA32UI : 4 entiers 32-bit par texel, soit 16 bytes — exactement la taille d'un splat packed.

Plusieurs SplatMesh peuvent coexister ; ils sont accumulés dans un WebGLArrayRenderTarget partagé (SplatAccumulator) qui attribue à chacun une plage de lignes dans la texture.

Points clés à retenir

Pas de .to(device)

Contrairement à PyTorch, il n'y a pas d'appel explicite pour envoyer les données au GPU. Tout passe par le flag needsUpdate = true que THREE.js surveille et qui déclenche l'upload WebGL au prochain frame.

Format RGBA32UI

Toutes les textures principales sont en RGBA32UI : 4 entiers 32-bit non signés par texel. C'est idéal pour stocker des splats packés (16 bytes = exactement 1 texel).

Workers pour ne pas bloquer

Le décodage WASM et le tri se font dans des Web Workers séparés. Le thread principal ne fait que gérer le rendu et la logique Three.js.

LoD par Range requests

SplatPager charge les chunks à la demande via des requêtes HTTP Range, permettant un chargement progressif sans avoir à télécharger tout le fichier d'un coup.
